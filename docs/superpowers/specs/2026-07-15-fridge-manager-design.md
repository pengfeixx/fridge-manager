# 冰箱食材管家 App — 设计文档

- **文档日期**：2026-07-15
- **状态**：待评审
- **目标平台**：Android（Flutter 单代码库，后期可扩 iOS/Web）
- **产品定位**：面向中国家庭的冰箱食材管理工具。以拍照识图为主入口录入食材，按保质期提醒、按快过期食材推荐菜谱，并依据《中国居民膳食指南》给出本周买菜建议。

---

## 1. 背景与目标

家庭买菜后常忘记冰箱里还有什么、什么时候过期，导致食材浪费或吃了不新鲜的食物；同时难以判断"本周吃得是否均衡、该再买什么"。本 App 解决：

1. **食材可见**：清楚知道冰箱里有什么、还有几天过期。
2. **减少浪费**：按快过期食材推荐菜谱，临期提醒。
3. **吃得更健康**：依据膳食指南分析本周饮食缺口，给出量化买菜建议。

### 1.1 成功标准

- 用户买完菜用拍照一次操作即可完成入库（含保质期默认值）。
- 每日首页能看到"今天该优先吃掉什么"的菜谱推荐。
- 能给出本周各类食物的缺口与购物清单。
- 第一期完全离线可用（AI 仅作为增强，离线/未配置时降级不崩溃）。

### 1.2 范围与分期

整体分四个阶段交付（详见第 5 节）。**第一期交付阶段一（核心循环，不依赖 AI）与阶段二（AI 接入）**，阶段三/四视后续推进。后端服务器、多账号同步本期**不做实现**，只在架构上预留接入点。

## 2. 总体架构与技术栈

### 2.1 技术选型

| 关注点 | 选型 | 理由 |
|---|---|---|
| 框架 | Flutter (Dart) | 单代码库，后期可扩 iOS/Web |
| 本地数据库 | drift (SQLite) | 类型安全、支持响应式 Stream、迁移机制成熟 |
| 轻量配置 | shared_preferences | 存 AI 配置等少量键值 |
| 状态管理 / DI | riverpod | Provider/override 机制便于替换实现与测试 |
| 路由 | go_router | 声明式路由，支持深链接 |
| 网络 | dio | 调用 OpenAI 兼容接口、上传图片 |
| 本地通知 | flutter_local_notifications | 临期提醒 |
| 图片选取 | image_picker | 拍照 / 相册 |
| 语音 | speech_to_text 或云端 STT | 语音录入食材 |

### 2.2 分层架构

```
┌─────────────────────────────────────────────┐
│  UI 层 (lib/features/*)  只关心展示与交互      │
├─────────────────────────────────────────────┤
│  Application 层  Riverpod Provider / 用例编排  │
├─────────────────────────────────────────────┤
│  Domain 层  实体、Repository 接口、AiService    │
│           接口、保质期/推荐/营养等领域逻辑（纯 Dart）│
├─────────────────────────────────────────────┤
│  Data 层 (drift 实现) / Service 层 (AI 实现)    │
└─────────────────────────────────────────────┘
```

### 2.3 扩展点设计（满足"本期纯本地，后期可接后端"）

- **Repository 抽象**：接口定义在 Domain 层（纯 Dart，无 Flutter 依赖），Data 层提供 `LocalXxxRepository`（drift）。后期新增 `RemoteXxxRepository`（HTTP）只需实现接口并通过 Riverpod `override` 切换绑定，UI/Application 层零改动。
- **AiService 抽象**：接口在 Domain 层，本期实现 `OpenAiCompatibleAiService`（直连云端 OpenAI 兼容协议）。后期可换 `BackendAiService`（走自有后端代理统一鉴权与计费）。
- **统一依赖注入**：全部通过 Riverpod 注入，测试时换成 mock 实现。
- **数据层可替换的代价**：drift 表结构设计需与未来后端 API 的数据契约保持一致，避免迁移期返工（见第 4 节数据模型）。

### 2.4 项目结构（feature-first）

```
lib/
  core/            # 通用工具、主题、错误处理、路由常量
  data/            # drift 数据库、DAO、LocalRepository 实现、seed 数据
  domain/          # 实体、Repository/AiService 接口、领域逻辑
  services/        # AiService 实现、保质期计算、推荐引擎、营养算法
  features/        # fridge / recipes / family / nutrition / settings / scan
  main.dart
  app.dart
test/              # 与 lib 结构对应的单元/Widget/集成测试
```

## 3. UX 原则

1. **拍照识图为主入口**：首页底部主导航以醒目"拍照"按钮为中心；买完菜一次拍照即可完成录入。语音为次入口；**手动录入为精简兜底路径**（仍完整支持名称/数量/存储位置/保质期手动填写），但 UI 上不喧宾夺主。
2. **最少操作完成高频任务**：入库、标记用完、看推荐都应在 1～2 次点击内完成。
3. **状态一目了然**：临期用颜色（红/黄/绿）直观表达，首页即"今天该吃什么"。
4. **降级友好**：AI 不可用时不阻断核心流程，给出明确引导。

## 4. 领域数据模型

drift 表与对应 domain 实体一一对应。下表列关键字段。

| 实体 | 关键字段 | 说明 |
|---|---|---|
| `FoodItem` 食材 | id, name, categoryId, quantity, unit, storage(冷藏/冷冻/常温), addedDate, shelfLifeDays, status(在库/已用/已弃), note | 冰箱里的一条食材；**expireDate 不入库**，展示时由 `addedDate + shelfLifeDays` 实时计算 |
| `FoodCategory` 类别 | id, name, icon, defaultShelfLifeDaysByStorage | 叶菜/根茎/禽肉/水产/蛋奶等 |
| `ShelfLifeRule` 保质期规则 | id, foodName, aliases[], storage, defaultDays | 内置参考表，用户可改；AI 识图按此填默认值 |
| `Recipe` 菜谱 | id, title, ingredients[], steps[], tags[], source(local/ai), note | 本地库 + AI 补充 |
| `RecipeIngredient` | recipeId, foodName, amount, unit | 菜谱与食材关联 |
| `FamilyMember` 家庭成员 | id, name, age, gender, dietaryTags[], allergies[] | 年龄/性别用于营养计算 |
| `MealLog` 饮食记录 | id, date, mealType(早/午/晚/加餐), recipeId?, foodEntries[] | **不记录哪些成员吃**，默认全家都吃 |
| `NutritionGuide` 膳食指南 | id, ageRangeMin, ageRangeMax, gender, foodCategory, dailyRecommendGram | 内置《指南(2022)》参考表 |

> 注：AI 配置（`AiConfig`：baseUrl / textModel / visionModel / voiceModel / presets）不进 drift，由 `shared_preferences` 存储明文项、`flutter_secure_storage` 加密存储 `apiKey`，封装为 `AiConfigRepository` 暴露给上层。

### 4.1 领域规则

- **过期计算**：`expireDate = addedDate + shelfLifeDays`，每次展示实时计算，避免跨天/时区偏差。
- **状态流转**：在库 →（剩余天数 ≤ 阈值）→ 临期提醒 →（过期）→ 已过期；用户可手动标记"已用完/已丢弃"。
- **推荐打分**：食材匹配度（菜谱所需食材已有覆盖率）+ 临近过期加权；忌口/过敏为硬过滤（任一成员忌口即过滤）。
- **营养计算口径**：家庭每日某类食物推荐量 = 各成员（按年龄/性别）推荐量之和；本周该类实际摄入 = 本周 `MealLog` 中该类食物汇总；缺口 = 推荐量 − 实际摄入 − 当前在库可用量（下限 0）。

## 5. 分阶段需求拆解

优先级：**P0 必做 / P1 应做 / P2 可选**。阶段一、二为本期交付。

### 阶段一 · 核心循环（不依赖 AI）

**1.1 食材管理 `features/fridge`**
- P0 手动添加食材：名称、类别、数量单位、存储位置、购买日期、保质期天数
- P0 编辑 / 删除食材
- P0 按存储位置（冷藏/冷冻/常温）分组列表
- P0 过期倒计时（红=已过期 / 黄=3天内 / 绿=安全）
- P0 标记"已用完/已丢弃"
- P1 搜索、按类别筛选、按到期日排序
- P1 内置保质期参考表 `ShelfLifeRule`，添加时自动带出默认天数，可改
- P2 批量操作、库存统计（消耗/丢弃量）

**1.2 菜谱与推荐 `features/recipes`**
- P0 本地菜谱库种子数据（`Recipe` + `RecipeIngredient`）
- P0 推荐：快过期食材优先 + 食材匹配度打分排序
- P0 忌口/过敏硬过滤（关联家庭成员）
- P0 菜谱详情：用料、步骤
- P1 手动添加 / 收藏菜谱
- P1 "用这道菜" → 扣减对应食材库存 + 写入 `MealLog`

**1.3 家庭成员 `features/family`**
- P0 录入成员：姓名、年龄、性别
- P0 忌口标签（素食/清真/不吃辣…）+ 过敏原（花生/海鲜/麸质…）
- P1 编辑 / 删除成员

**1.4 提醒**
- P1 本地通知：每日临期食材提醒（可配提前天数）

### 阶段二 · AI 接入（增强录入与菜谱）

**2.1 AI 配置页 `features/settings`**
- P0 配置 Base URL / API Key / 文本模型 / 视觉模型 / 语音模型
- P0 API Key 加密存储（flutter_secure_storage）
- P0 "测试连接"按钮
- P1 预设快捷选项（豆包/DeepSeek/OpenAI…）+ 自定义

**2.2 拍照识图 `features/scan`（主入口）**
- P0 调用视觉模型识别照片中食材清单（名称、估测数量）
- P0 结果可在确认页逐条核对（名称/数量/存储位置/保质期默认值可改）→ 批量入库
- P1 识别购物小票辅助录入
- P2 调用失败 / 未配置时降级为纯手动（带引导提示）

**2.3 语音输入**
- P0 语音转文字 → 文本模型解析为食材清单 → 确认入库
- P1 连续语音录入

**2.4 AI 菜谱补充**
- P1 本地库匹配不到时，按现有食材让 AI 生成菜谱并入库（source=ai）

### 阶段三 · 营养买菜建议

**3.1 饮食记录 `features/nutrition`**
- P0 记录每餐吃了什么（关联菜谱或自由记录食材）
- P0 本周饮食日历视图

**3.2 营养分析**
- P0 内置《中国居民膳食指南(2022)》参考表（年龄段 × 性别 × 食物类别每日推荐量）
- P0 算法：家庭年龄结构 → 家庭每日/每周推荐量 → 对比本周实际摄入 → 各类别缺口/盈余

**3.3 买菜建议**
- P0 输出量化购物清单（如"叶菜还缺约 600g/周"），结合当前库存
- P1 AI 把量化缺口润色为自然语言购物建议

### 阶段四 · 扩展预留（本期仅留接口，不实现）

- P2 数据导出 / 导入（JSON 备份）
- P2 Repository 切换为远程实现的接入点、账号登录界面占位
- P2 多设备同步（依赖后端）

## 6. 关键流程

### 流程 A · 拍照入库（主路径）
相机/相册取图 → 压缩 → 视觉模型（OpenAI 兼容，image + prompt，要求返回结构化 JSON 食材清单）→ 解析结果 → 用户在确认页逐条核对（名称/数量/存储位置/保质期，默认值来自 `ShelfLifeRule`，可改）→ 批量写 `FoodItem`。

### 流程 B · 菜谱推荐（每日首页）
扫描在库 `FoodItem` → 计算各食材剩余天数 → 推荐引擎打分（临近过期权重 + 菜谱食材已有覆盖率）→ 过滤家庭成员忌口/过敏 → 排序输出 Top N。

### 流程 C · 营养买菜建议
读取家庭成员 → 家庭每日各类别推荐量 ×7 → 汇总本周 `MealLog` 各类别实际摄入 → 差值即缺口 → 减去当前在库可用量 → 输出购物清单（可选 AI 润色）。

### 流程 D · 保质期处理
添加食材时查 `ShelfLifeRule`（按名称/别名匹配 + 存储位置）→ 带出默认天数 → `expireDate` 展示时由 `addedDate + shelfLifeDays` 实时算 → 每日定时任务刷新状态 + 触发通知。

### 流程 E · 语音入库
语音 → 转文字 → 文本模型解析为结构化食材清单 → 复用流程 A 的确认页核对 → 入库。

### 错误与降级
- AI 调用失败：识图/语音降级为手动；菜谱回退本地库；UI 明确提示而非崩溃。
- 无网络 / 未配置 AI：阶段一功能完全可用；阶段二功能给出"去配置"引导。

## 7. 非功能需求

- **性能**：首页冷启动到可交互 ≤ 2s；菜谱推荐本地计算 ≤ 200ms；AI 请求带 loading 与超时（默认 30s）。
- **隐私**：API Key 加密存储；图片仅用于识别，识别完不持久化原图（可选保留缩略图）。
- **可访问性**：遵循 Material 3，文字支持系统字号；关键操作有文字标签。
- **国际化**：默认中文，文案走 `arb` 资源，预留多语言结构。
- **测试**：领域逻辑（保质期/推荐/营养算法）单元测试全覆盖；Repository 用内存实现做契约测试；关键流程 Widget 测试；阶段二 AI 用 mock `AiService`。
- **可维护性**：领域层零 Flutter 依赖，便于纯 Dart 测试与未来抽离。

## 8. 风险与对策

| 风险 | 对策 |
|---|---|
| AI 识图数量/单位不准 | 确认页强制人工核对后再入库；保留可编辑 |
| OpenAI 兼容协议各家差异 | 配置页支持自定义 Base URL/模型名；"测试连接"校验 |
| 膳食指南量化粗糙 | 明确标注为"参考"，按食物类别粗粒度而非逐菜品精确营养 |
| 本期无同步，换手机丢数据 | 阶段四补 JSON 导出导入作最低保障 |
| 未来接后端返工 | Repository/AiService 接口先行，数据契约与后端 API 对齐 |
